# 開発ルール・GitHub運用ルール

## 1. 目的

本ドキュメントは、Reserve CRM の開発におけるGitHub運用ルール、ブランチ戦略、Issue管理、Pull Request運用、コミット規約、レビュー観点を定義することを目的とする。

本プロジェクトは個人開発であるが、実務プロジェクトを想定し、チーム開発でも通用する開発ルールを採用する。

---

## 2. 基本方針

Reserve CRM では、以下の方針で開発を進める。

* すべての作業はIssueで管理する
* 実装はIssue単位でブランチを作成する
* `main` ブランチへ直接コミットしない
* Pull Request を作成してからマージする
* Pull Request では変更内容、確認内容、関連Issueを明記する
* CIが通過した状態でマージする
* 実装に合わせてドキュメントも更新する

---

## 3. ブランチ戦略

## 3.1 使用するブランチ

| ブランチ         | 用途                  |
| ------------ | ------------------- |
| `main`       | 本番想定環境にデプロイする安定ブランチ |
| `develop`    | 開発統合ブランチ            |
| `feature/*`  | 機能開発ブランチ            |
| `fix/*`      | 不具合修正ブランチ           |
| `chore/*`    | 設定変更、依存関係更新など       |
| `docs/*`     | ドキュメント作成・修正         |
| `refactor/*` | 振る舞いを変えない内部改善       |

---

## 3.2 ブランチ命名規則

ブランチ名は以下の形式とする。

```text
種類/Issue番号-概要
```

### 例

```text
feature/BL-001-login
feature/BL-011-create-customer
feature/BL-016-create-reservation
fix/BL-022-reservation-overlap-check
docs/SP0-012-development-rule
chore/setup-github-actions
```

---

## 3.3 ブランチ運用

原則として、以下の流れで開発する。

```text
develop から作業ブランチを作成
↓
作業ブランチで実装
↓
Pull Request を作成
↓
CI確認
↓
レビュー
↓
develop へマージ
↓
リリースタイミングで main へマージ
```

MVP期間中は、`develop` を開発の中心ブランチとし、安定したタイミングで `main` に反映する。

# 4. Issue運用

## 4.1 Issueの種類

| 種別    | 内容               |
| ----- | ---------------- |
| Epic  | 大きな機能領域          |
| Story | ユーザー価値単位の機能      |
| Task  | 実装、設計、調査などの作業    |
| Bug   | 不具合              |
| Chore | 開発環境、CI、依存関係更新など |
| Doc   | ドキュメント作成・更新      |

---

## 4.2 Issue作成ルール

Issueには以下を記載する。

* タイトル
* 種別
* 対応Epic
* 背景
* 実施内容
* 受け入れ条件
* 関連ドキュメント
* 対応予定スプリント
* 優先度

---

## 4.3 Issueテンプレート

```md
## 概要

## 背景

## 対応内容

- [ ] 
- [ ] 
- [ ] 

## 受け入れ条件

- [ ] 
- [ ] 

## 関連Issue / PR

## 関連ドキュメント

## 優先度

Must / Should / Could / Won't

## 対応スプリント

Sprint X
```

---

## 4.4 Issueステータス

Issueは以下の状態で管理する。

| ステータス       | 内容                |
| ----------- | ----------------- |
| Backlog     | 未着手               |
| Ready       | 着手可能              |
| In Progress | 作業中               |
| Review      | Pull Requestレビュー中 |
| Done        | 完了                |
| Blocked     | 何らかの理由で停止中        |

---

# 5. Pull Request運用

## 5.1 Pull Request作成ルール

機能追加、修正、ドキュメント更新は、原則としてPull Requestを作成して反映する。

Pull Requestには以下を記載する。

* 概要
* 対応Issue
* 変更内容
* 確認方法
* スクリーンショット
* 影響範囲
* 未対応事項

---

## 5.2 Pull Requestテンプレート

```md
## 概要

## 対応Issue

Closes #

## 変更内容

- 
- 
- 

## 確認内容

- [ ] ローカルで動作確認した
- [ ] Lint が通る
- [ ] 型チェックが通る
- [ ] テストが通る
- [ ] 画面表示を確認した
- [ ] 必要なドキュメントを更新した

## スクリーンショット

必要に応じて添付する。

## 影響範囲

## 未対応事項

## 備考
```

---

## 5.3 Pull Requestの粒度

Pull Requestは、大きすぎない単位に分割する。

### 良い例

* 顧客登録APIを追加する
* 顧客登録フォームを追加する
* 予約重複チェックのserviceを追加する
* 請求金額計算のテストを追加する

### 避ける例

* 顧客管理を全部実装する
* 予約管理を全部実装する
* MVPを全部作る

# 6. コミットメッセージ規約

## 6.1 基本形式

コミットメッセージは以下の形式とする。

```text
type(scope): summary
```

### type一覧

| type     | 内容                |
| -------- | ----------------- |
| feat     | 新機能追加             |
| fix      | 不具合修正             |
| docs     | ドキュメント修正          |
| style    | フォーマット修正          |
| refactor | 振る舞いを変えないリファクタリング |
| test     | テスト追加・修正          |
| chore    | 設定変更、依存関係更新など     |
| ci       | CI設定変更            |
| build    | ビルド設定変更           |

---

## 6.2 コミット例

```text
feat(auth): add login page
feat(customers): add customer create form
feat(reservations): add overlap validation
fix(reservations): exclude cancelled reservations from overlap check
test(invoices): add invoice amount calculation tests
docs(project): add development rule
chore(prisma): add initial schema
ci(github): add pull request workflow
```

---

## 6.3 コミット単位

コミットは、意味のある小さな単位で行う。

### 良い例

* Prisma schema に customers テーブルを追加する
* 顧客登録フォームを追加する
* 顧客登録バリデーションを追加する
* 顧客登録のテストを追加する

### 避ける例

* いろいろ修正
* 作業中
* 一旦コミット
* 全部追加

# 7. コードレビュー観点

個人開発であっても、Pull Request作成時にセルフレビューを行う。

## 7.1 機能観点

* 受け入れ条件を満たしているか
* 画面から期待通りに操作できるか
* 異常系の挙動が考慮されているか
* エラーメッセージが分かりやすいか

---

## 7.2 セキュリティ観点

* 認証が必要な画面・処理に認証チェックがあるか
* 権限が必要な処理にroleチェックがあるか
* パスワードや秘密情報をログに出していないか
* 環境変数をリポジトリに含めていないか
* 顧客情報を不要に露出していないか

---

## 7.3 データ整合性観点

* 予約重複チェックが正しく行われているか
* 請求金額が正しく計算されているか
* 支払い済み登録後に売上へ反映されるか
* 論理削除データを誤って表示していないか
* 店舗IDによるデータ分離が考慮されているか

---

## 7.4 保守性観点

* UI、service、repository の責務が分かれているか
* 同じ処理を重複実装していないか
* 型定義が適切か
* 関数名、変数名が分かりやすいか
* 複雑な処理に最低限のコメントがあるか

---

## 7.5 テスト観点

* 重要な業務ロジックにテストがあるか
* 正常系だけでなく異常系も確認しているか
* 予約重複判定の境界値を確認しているか
* 請求金額計算のテストがあるか
* 既存テストが通るか

---

# 8. Definition of Ready

Issueは以下を満たした場合に着手可能とする。

* 目的が明確である
* 受け入れ条件が記載されている
* 優先度が決まっている
* 対応スプリントが決まっている
* 依存するIssueが明確である
* 必要な画面・API・DBの概要が分かっている

---

# 9. Definition of Done

Issueは以下を満たした場合に完了とする。

* 実装またはドキュメント作成が完了している
* 受け入れ条件を満たしている
* Lint が通っている
* 型チェックが通っている
* 必要なテストが通っている
* Pull Request が作成されている
* セルフレビューが完了している
* 必要なドキュメントが更新されている
* `develop` にマージされている

---

# 10. CIで確認する項目

Pull Request作成時に、GitHub Actionsで以下を確認する。

| 項目         | コマンド例            |
| ---------- | ---------------- |
| 依存関係インストール | `pnpm install`   |
| Lint       | `pnpm lint`      |
| 型チェック      | `pnpm typecheck` |
| 単体テスト      | `pnpm test`      |
| ビルド        | `pnpm build`     |

MVP後半では、必要に応じて以下も追加する。

| 項目              | コマンド例                  |
| --------------- | ---------------------- |
| E2Eテスト          | `pnpm test:e2e`        |
| Prisma schema検証 | `pnpm prisma validate` |
| フォーマット確認        | `pnpm format:check`    |

# 11. ドキュメント更新ルール

以下に該当する変更を行った場合は、関連ドキュメントも更新する。

| 変更内容    | 更新対象                                 |
| ------- | ------------------------------------ |
| 機能追加    | product-backlog.md / user-stories.md |
| 画面追加    | screen-list.md / screen-flow.md      |
| DB変更    | er-draft.md / table-definition.md    |
| API追加   | api-list.md                          |
| 権限変更    | role-permission.md                   |
| スプリント完了 | sprint-xx.md                         |
| リリース    | release-note.md                      |

# 13. 環境変数管理

環境変数は `.env` に定義する。

`.env` はGit管理しない。

代わりに `.env.example` を用意し、必要な環境変数名のみを共有する。

## 例

```text
DATABASE_URL=
AUTH_SECRET=
NEXTAUTH_URL=
SENTRY_DSN=
```

---

# 14. セキュリティルール

開発時は以下を守る。

* パスワードを平文で保存しない
* `.env` をコミットしない
* APIキーやDB接続情報をコードに直書きしない
* 顧客情報をログに出力しない
* サーバー側で認可チェックを行う
* 入力値は必ずバリデーションする

---

# 15. スプリント運用

各スプリントでは、以下を記録する。

## スプリント開始時

* スプリントゴール
* 対象バックログ
* 実施予定タスク
* リスク

## スプリント中

* 実装Issue
* Pull Request
* 発生した問題
* 仕様変更・追加要望

## スプリント終了時

* 完了したIssue
* 未完了のIssue
* デモ内容
* レビュー結果
* レトロスペクティブ
* 次スプリントへの改善点

---

# 16. レトロスペクティブ観点

各スプリント終了時に、以下を記録する。

## Keep

継続したいこと。

## Problem

問題だったこと。

## Try

次に試すこと。

---

# 17. MVP期間中の判断ルール

MVP期間中は、以下を優先する。

1. 顧客登録から予約、請求、売上確認までの主要業務フロー
2. 予約重複防止
3. 請求金額の正確性
4. 認証・権限の基本制御
5. テストによる重要ロジックの保護

以下は、MVP完了後の改善候補とする。

* 見た目の細かい作り込み
* 高度な分析
* 外部サービス連携
* 請求書PDF
* Web予約フォーム
* LINE連携
* 複数店舗対応

---

# 18. 備考

本ルールは初期版であり、開発を進める中で必要に応じて更新する。

ただし、以下の原則は維持する。

* Issueで作業を管理する
* Pull Requestを通して変更する
* CIを通した状態でマージする
* 重要な仕様変更はドキュメントに反映する
