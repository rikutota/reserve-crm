# スコープ定義

## 1. 目的

本ドキュメントは、Reserve CRM のMVPにおける開発対象範囲を定義することを目的とする。

本プロジェクトでは、小規模美容サロンが日々の予約、顧客、請求、売上確認を行うために必要な最小限の機能をMVPとして開発する。

MVPでは、店舗スタッフが利用する管理画面を中心に開発し、外部サービス連携や高度な分析機能は対象外とする。

---

## 2. 開発対象の業務範囲

MVPでは、以下の業務範囲を対象とする。

| 業務領域   | 対象範囲                 |
| ------ | -------------------- |
| 認証     | 管理画面へのログイン、ログアウト     |
| スタッフ管理 | スタッフ情報の登録、編集、無効化     |
| メニュー管理 | 施術メニューの登録、編集、無効化     |
| 顧客管理   | 顧客情報の登録、検索、編集、詳細確認   |
| 予約管理   | 予約登録、予約一覧、予約変更、キャンセル |
| 予約制御   | 同一スタッフの予約重複防止        |
| 請求管理   | 来店済み予約からの簡易請求作成      |
| 支払い管理  | 支払い済み登録、未払い確認        |
| 売上管理   | 日次売上、月次売上の確認         |
| 運用補助   | 基本的なエラー表示、最低限の操作ログ   |

---

## 3. ユーザー範囲

MVPでは、以下のユーザーを対象とする。

### 3.1 オーナー

店舗の管理者として、以下の操作を行う。

* スタッフ管理
* メニュー管理
* 顧客管理
* 予約管理
* 請求管理
* 売上確認

### 3.2 スタッフ

店舗の現場担当者として、以下の操作を行う。

* 顧客登録
* 顧客検索
* 予約登録
* 予約変更
* 予約キャンセル
* 来店後の請求処理
* 支払い済み登録

---

## 4. 機能スコープ

### 4.1 認証機能

#### 対象

* ログイン
* ログアウト
* 認証状態の保持
* 未ログインユーザーのアクセス制御

#### 対象外

* パスワードリセット
* メール認証
* 二要素認証
* 外部アカウントログイン

---

### 4.2 スタッフ管理

#### 対象

* スタッフ登録
* スタッフ一覧表示
* スタッフ編集
* スタッフ無効化
* 役割の設定

#### 対象外

* 勤怠管理
* シフト管理
* 給与管理
* スタッフ評価管理

---

### 4.3 メニュー管理

#### 対象

* メニュー登録
* メニュー一覧表示
* メニュー編集
* メニュー無効化
* 所要時間の設定
* 料金の設定

#### 対象外

* キャンペーン価格管理
* クーポン管理
* セットメニュー管理
* 在庫連動

---

### 4.4 顧客管理

#### 対象

* 顧客登録
* 顧客一覧表示
* 顧客検索
* 顧客詳細表示
* 顧客編集
* 顧客メモ管理
* 注意事項管理

#### 対象外

* 顧客ランク自動判定
* ポイント管理
* 顧客向けマイページ
* 顧客への自動メール配信
* 顧客データの外部CRM連携

---

### 4.5 予約管理

#### 対象

* 予約登録
* 予約一覧表示
* 予約詳細表示
* 予約変更
* 予約キャンセル
* 予約ステータス変更
* 同一スタッフの予約重複チェック
* メニュー所要時間による終了時刻計算

#### 対象外

* 顧客自身によるWeb予約
* LINEからの自動予約
* Instagram DMとの連携
* キャンセル待ち
* リマインド通知
* 複数店舗をまたぐ予約管理

---

### 4.6 請求・支払い管理

#### 対象

* 予約から請求作成
* 請求一覧表示
* 請求詳細表示
* 支払い済み登録
* 未払い確認
* 支払い方法の登録

#### 対象外

* クレジットカード決済
* QR決済連携
* 銀行振込消込
* 請求書PDF発行
* 領収書PDF発行
* 会計ソフト連携

---

### 4.7 売上管理

#### 対象

* 日次売上一覧
* 月次売上一覧
* 未払い件数表示
* 支払い済み金額の集計

#### 対象外

* 高度な売上分析
* スタッフ別詳細分析
* メニュー別詳細分析
* グラフによる高度な可視化
* 経営レポート自動生成

---

## 5. データスコープ

MVPで扱う主なデータは以下とする。

| データ  | 内容                   |
| ---- | -------------------- |
| ユーザー | ログインする利用者情報          |
| スタッフ | 予約担当者情報              |
| メニュー | 施術メニュー情報             |
| 顧客   | 来店顧客情報               |
| 予約   | 顧客、スタッフ、メニュー、日時の予約情報 |
| 請求   | 予約に紐づく請求情報           |
| 支払い  | 請求に対する支払い情報          |
| 売上   | 支払い済み請求をもとにした集計情報    |

---

## 6. 対応デバイス

MVPでは、以下の利用環境を想定する。

### 対象

* PCブラウザ
* タブレットブラウザ
* スマートフォンブラウザ

### 優先度

1. PCブラウザ
2. タブレットブラウザ
3. スマートフォンブラウザ

MVPでは、まずPCブラウザでの業務利用を優先する。

スマートフォン対応はレスポンシブ表示の基本対応までとし、スマホ専用UIの作り込みは後続対応とする。

---

## 7. 非機能スコープ

MVPでは、以下の非機能要件を対象とする。

| 項目      | MVPでの方針                   |
| ------- | ------------------------- |
| セキュリティ  | 認証済みユーザーのみ管理画面を利用可能にする    |
| パスワード管理 | パスワードはハッシュ化して保存する         |
| 通信      | 本番想定環境ではHTTPSを使用する        |
| 権限      | オーナーとスタッフの基本的な役割を分ける      |
| エラー処理   | 入力エラー、認証エラー、権限エラーを適切に表示する |
| ログ      | 重要操作を最低限記録する              |
| バックアップ  | 本番想定環境ではDBバックアップ方針を用意する   |
| テスト     | 主要な業務ロジックに自動テストを用意する      |

---

## 8. MVP完了時の業務フロー

MVP完了時点で、以下の業務フローを実行できる状態を目指す。

1. オーナーまたはスタッフがログインする
2. スタッフ情報を登録する
3. メニュー情報を登録する
4. 顧客情報を登録する
5. 顧客、スタッフ、メニュー、日時を指定して予約を登録する
6. システムが予約重複をチェックする
7. 予約一覧で予約状況を確認する
8. 来店後、予約ステータスを来店済みに変更する
9. 予約内容をもとに請求を作成する
10. 支払い済みに変更する
11. 支払い済み金額が日次売上に反映される

---

## 9. スコープ変更の扱い

MVP開発中に新しい要望が出た場合、即時実装は行わず、プロダクトバックログに追加する。

追加された要望は、以下の観点で優先度を判断する。

* MVPの成功に必要か
* 現在の業務課題を直接解決するか
* 実装コストが大きすぎないか
* 既存機能への影響が大きくないか
* リリース時期に影響しないか

MVPに必須でない要望は、原則としてMVPリリース後の改善候補とする。
