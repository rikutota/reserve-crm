# 初期画面一覧

## 1. 目的

本ドキュメントは、Reserve CRM のMVPで開発する画面一覧を定義することを目的とする。

画面一覧を作成することで、MVPに必要なUI範囲を明確にし、スプリントごとの画面実装計画、画面遷移設計、API設計、テスト設計の前提とする。

---

## 2. 画面設計方針

MVPでは、店舗スタッフが日常業務で使う管理画面を中心に開発する。

画面設計では、以下を重視する。

* 業務フローに沿って迷わず操作できること
* 顧客登録、予約登録、請求処理までを少ない手順で行えること
* 予約状況や未払い状況をすぐ確認できること
* PCブラウザでの利用を優先すること
* スマートフォンでは基本的な閲覧・操作ができるレスポンシブ対応を行うこと

---

## 3. 画面カテゴリ

MVPで扱う画面カテゴリは以下とする。

| カテゴリ    | 内容                   |
| ------- | -------------------- |
| 認証      | ログイン、ログアウト           |
| ダッシュボード | 今日の予約、売上、未払いの概要確認    |
| スタッフ管理  | スタッフの登録・編集・一覧        |
| メニュー管理  | 施術メニューの登録・編集・一覧      |
| 顧客管理    | 顧客の登録・検索・詳細・編集       |
| 予約管理    | 予約の登録・一覧・詳細・変更・キャンセル |
| 請求管理    | 請求の作成・一覧・支払い済み登録     |
| 売上管理    | 日次・月次売上の確認           |
| 設定      | 店舗情報や基本設定            |
| エラー     | 認証エラー、権限エラー、404など    |

# 5. 各画面詳細

## SCR-001：ログイン画面

### 目的

ユーザーがメールアドレスとパスワードでログインする。

### 主な利用者

* オーナー
* スタッフ

### 主な機能

* メールアドレス入力
* パスワード入力
* ログイン実行
* 認証エラー表示

### 主な表示項目

* メールアドレス
* パスワード
* ログインボタン
* エラーメッセージ

### 備考

MVPでは、パスワードリセット、メール認証、二要素認証は対象外とする。

# 8. 画面設計上の注意点

## 8.1 業務フローを優先する

MVPでは、見た目の作り込みよりも、以下の業務フローがスムーズに行えることを優先する。

```text
顧客登録
↓
予約登録
↓
来店済み
↓
請求作成
↓
支払い済み
↓
売上確認
```

## 8.2 一覧画面から詳細画面へ遷移できるようにする

顧客、予約、請求は、一覧から詳細を確認できる設計とする。

## 8.3 登録画面と編集画面のUIは可能な限り共通化する

スタッフ、メニュー、顧客、予約の登録・編集画面は、入力フォームの再利用を前提に設計する。

## 8.4 エラーは画面上で分かりやすく表示する

入力エラー、認証エラー、予約重複エラー、権限エラーは、ユーザーが次に何をすればよいか分かる文言で表示する。

## 8.5 モバイル対応は基本対応までとする

MVPではPCブラウザ利用を優先し、スマートフォンでは最低限の閲覧・登録操作ができることを目標とする。
